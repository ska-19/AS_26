# Практическая работа: «Умный ассистент с характером»

**Модуль 2 — LangChain: от основ к production-ready системам**

---

## Что вы создадите

Вы напишете **интерактивного чат-бота**, который работает прямо в терминале (CLI). Это не просто обёртка над LLM — бот будет **думать перед ответом**: сначала определять тип запроса, затем выбирать подходящую стратегию ответа, и при этом помнить, о чём вы говорили раньше.

Конечный результат выглядит так:

```
🤖 Умный ассистент с характером
Характер: friendly | Память: buffer
────────────────────────────────

> Привет! Меня зовут Алексей
[small_talk] Привет, Алексей! Рад знакомству! Чем могу помочь?
confidence: 0.95 | tokens: ~42

> Что такое LCEL?
[question] LCEL — это LangChain Expression Language, декларативный синтаксис
для описания цепочек обработки данных через оператор |
confidence: 0.91 | tokens: ~68

> /character sarcastic
✓ Характер изменён на: sarcastic

> Как меня зовут?
[question] Ну вы же только что представились — Алексей. Память у меня
получше, чем у некоторых. 😏
confidence: 0.93 | tokens: ~35
```

Обратите внимание: бот **запомнил** имя из первого сообщения, **переключил стиль** после команды, и каждый ответ сопровождается **метаданными** (тип запроса, уверенность).

## Часть 1. Структурированные модели данных (Pydantic)

### Что нужно сделать

Определите три Pydantic-модели, которые будут описывать «язык» вашего бота:

**`RequestType`** — перечисление (Enum) возможных типов запроса:
- `question` — вопрос, требующий информации («Что такое Python?», «Как работает GIL?»)
- `task` — просьба что-то сделать («Напиши стих», «Расскажи анекдот»)
- `small_talk` — приветствие и болтовня («Привет!», «Как дела?»)
- `complaint` — жалоба, недовольство («Это ужасно работает!», «Почему так долго?»)
- `unknown` — бессмыслица или нераспознанный запрос («asdfghjkl»)

**`Classification`** — результат работы классификатора:
- `request_type` — один из типов выше
- `confidence` — число от 0 до 1, насколько модель уверена в классификации
- `reasoning` — строка, краткое обоснование (зачем? чтобы при отладке понимать, почему бот решил именно так)

**`AssistantResponse`** — итоговый ответ бота:
- `content` — текст ответа
- `request_type` — какой тип запроса был определён
- `confidence` — уверенность
- `tokens_used` — приблизительное число токенов (для мониторинга)

### Зачем это нужно

Вместо того чтобы работать с сырыми строками, вы сразу получаете **типизированные объекты** с валидацией. Если модель вернёт confidence = 1.5, Pydantic выбросит ошибку. Это паттерн, который в production спасает от неожиданных ответов LLM.

### Подсказка

```python
from pydantic import BaseModel, Field
from enum import Enum

class RequestType(str, Enum):
    QUESTION = "question"
    TASK = "task"
    # ... остальные типы

class Classification(BaseModel):
    request_type: RequestType = Field(description="Тип запроса")
    confidence: float = Field(ge=0, le=1, description="Уверенность")
    reasoning: str = Field(description="Обоснование")
```

## Часть 2. Классификатор запросов

### Что нужно сделать

Создайте LCEL-цепочку, которая принимает текст пользователя и возвращает объект `Classification`.

Цепочка выглядит так:

```
вход (строка) → подстановка в промпт → модель → PydanticOutputParser → Classification
```

В промпте классификатора должны быть:
1. **Описание каждого типа** — чтобы модель понимала, что есть что
2. **Few-shot примеры** — 2–3 примера на каждый тип («Привет!» → small_talk, «Что такое Python?» → question и т.д.)
3. **Инструкции формата** — `PydanticOutputParser` сам генерирует описание нужного JSON-формата, вы подставляете его через `{format_instructions}`

### Важный нюанс: обработка ошибок

Модель иногда возвращает невалидный JSON. Оберните вызов цепочки в try/except и в случае ошибки возвращайте fallback:

```python
Classification(
    request_type=RequestType.UNKNOWN,
    confidence=0.5,
    reasoning="Ошибка парсинга ответа модели"
)
```

### Подсказка

```python
from langchain_core.output_parsers import PydanticOutputParser
from langchain_core.runnables import RunnablePassthrough

parser = PydanticOutputParser(pydantic_object=Classification)

prompt = ChatPromptTemplate.from_messages([
    ("system", """Ты — классификатор. Определи тип запроса.
    ...примеры...
    {format_instructions}"""),
    ("human", "Запрос: {query}")
])

chain = (
    {"query": RunnablePassthrough(), "format_instructions": lambda _: parser.get_format_instructions()}
    | prompt
    | model
    | parser
)
```

Обратите внимание на конструкцию `{"query": ..., "format_instructions": ...}` — это `RunnableParallel`, который формирует словарь для промпта из разных источников.

## Часть 3. Обработчики и роутинг

### Что нужно сделать

Создайте **отдельный обработчик** для каждого типа запроса. Каждый обработчик — это своя LCEL-цепочка `prompt → model → StrOutputParser()` со своим системным промптом:

- **question** → «Дай информативный и полезный ответ. Если не знаешь — честно скажи»
- **task** → «Пользователь просит выполнить задачу. Сделай качественно»
- **small_talk** → «Поддержи беседу, будь приветлив. Если представился — запомни имя»
- **complaint** → «Прояви эмпатию, попробуй понять проблему и предложить решение»
- **unknown** → «Запрос непонятен. Вежливо попроси уточнить»

Затем реализуйте **роутинг** — механизм, который направляет запрос в нужный обработчик на основе результата классификации. Используйте `RunnableBranch` или простой словарь обработчиков с `dict.get()`.

### Зачем отдельные обработчики

Можно было бы запихнуть всё в один огромный промпт: «если вопрос — ответь так, если жалоба — ответь так...». Но разделение даёт:
- **Независимую настройку** каждой ветки (можно менять промпт для жалоб, не трогая остальное)
- **Разные модели** для разных типов (дешёвая модель для small talk, мощная для вопросов)
- **Чистую архитектуру** — каждый обработчик тестируется отдельно

## Часть 4. «Характеры» ассистента

### Что нужно сделать

Создайте словарь с несколькими «характерами» — наборами системных промптов, которые задают стиль общения:

- **friendly** — дружелюбный и позитивный, может использовать эмодзи
- **professional** — деловой, сдержанный, без панибратства
- **sarcastic** — с лёгкой иронией, но по делу
- **pirate** — говорит как пират («Арр!», «Тысяча чертей!», называет пользователя «матрос»)

Характер подставляется в системный промпт **каждого обработчика**. При смене характера командой `/character pirate` все обработчики пересоздаются с новым промптом.

### Подсказка

```python
CHARACTER_PROMPTS = {
    "friendly": "Ты — дружелюбный ассистент. Общаешься тепло...",
    "sarcastic": "Ты — саркастичный ассистент с юмором...",
    # ...
}

# В каждом обработчике:
system_prompt = CHARACTER_PROMPTS[character] + "\n\nЭто вопрос пользователя. Дай полезный ответ."
```

## Часть 5. Память диалога

### Что нужно сделать

Реализуйте класс `MemoryManager`, который хранит историю сообщений и передаёт её в каждый вызов модели.

**Стратегия «buffer»** — самая простая. Хранит последние N сообщений (например, 20). Когда их становится больше — старые отбрасываются. Плюс: простота. Минус: теряется контекст из начала разговора.

**Стратегия «summary»** — когда сообщений становится слишком много, старые сообщения отправляются в LLM с просьбой «сделай краткое содержание этого диалога». Саммари сохраняется и подставляется в начало контекста вместо полной истории. Плюс: помнит ключевые факты из длинного разговора. Минус: тратит токены на суммаризацию.

В промптах обработчиков используйте `MessagesPlaceholder(variable_name="history")` для подстановки истории.

### Проверка работы памяти

Бот должен корректно отвечать на такой диалог:

```
> Привет, меня зовут Даша
[small_talk] Привет, Даша! ...

> Мой любимый язык — Python
[small_talk] Отличный выбор! ...

> Как меня зовут и какой мой любимый язык?
[question] Вас зовут Даша, и ваш любимый язык — Python.
```

## Часть 6. CLI-интерфейс

### Что нужно сделать

Напишите интерактивный цикл `while True` с `input("> ")`, который:

1. Читает ввод пользователя
2. Если ввод начинается с `/` — обрабатывает как команду
3. Иначе — отправляет в `SmartAssistant.process()` и выводит результат

**Обязательные команды:**
- `/clear` — очистить историю диалога
- `/character <name>` — сменить характер (friendly, professional, sarcastic, pirate)
- `/memory <strategy>` — сменить стратегию памяти (buffer, summary)
- `/status` — показать текущие настройки и количество сообщений в истории
- `/help` — справка по командам
- `/quit` — выход

**Формат вывода ответа:**

```
[тип_запроса] Текст ответа
confidence: 0.92 | tokens: ~78
```

### Запуск с аргументами

Используйте `argparse` для параметров командной строки:

```bash
python smart_assistant.py                           # по умолчанию
python smart_assistant.py --character sarcastic      # с выбранным характером
python smart_assistant.py --memory summary           # с суммаризацией памяти
python smart_assistant.py --model gpt-5-mini             # с конкретной моделью
```

## Что ещё?

### 1. Потоковый вывод (streaming)
Вместо ожидания полного ответа выводите текст по мере генерации. Используйте метод `.stream()` вместо `.invoke()` и печатайте токены через `print(chunk, end="", flush=True)`. Пользователь увидит, как ответ «печатается» в реальном времени — как в ChatGPT.

### 2. Кэширование ответов
Если пользователь задаёт один и тот же вопрос повторно, незачем тратить токены. Подключите `InMemoryCache` или `SQLiteCache` из LangChain:
```python
from langchain_core.globals import set_llm_cache
from langchain_core.caches import InMemoryCache
set_llm_cache(InMemoryCache())
```

### 3. Fallback на запасную модель
Если основная модель (gemini-3-flash-preview, gpt-5-mini) недоступна или выдала ошибку, цепочка автоматически переключается на резервную модель (qwen3-32b). Используйте метод `.with_fallbacks()`:
```python
main_model = ChatOpenAI(model="gpt-5-mini")
fallback_model = ChatOpenAI(model="qwen3-32b")
resilient_model = main_model.with_fallbacks([fallback_model])
```

### 4. Entity Memory
Извлекайте из разговора именованные сущности (имена, города, предпочтения) и храните их в отдельном хранилище. Это позволит боту помнить факты даже после `/clear`.

### 5. Callbacks для метрик
Подключите LangChain Callbacks для отслеживания реального количества токенов, времени ответа и стоимости каждого запроса.

## Критерии оценки

| Критерий | Баллы | На что смотрим |
|---|---|---|
| Классификация работает корректно | 1 | Правильно определяет тип для разных запросов |
| Роутинг направляет на правильные обработчики | 1 | Каждый тип обрабатывается своей цепочкой |
| Структурированный вывод с Pydantic | 1 | Ответ — валидный объект, а не сырая строка |
| Память сохраняет контекст | 1 | Бот помнит имя, факты из ранних сообщений |
| CLI-интерфейс удобен | 1 | Команды работают, вывод читаемый |
| Расширения | 1+1+1+1 | Стриминг, кэш, fallback, entity memory |
| Код чистый и документирован | 1 | Функции с docstring, логичная структура |
| Тесты | 1 | Хотя бы 3–5 тестов на ключевые сценарии |

**Итого:** 10 баллов + 1 (тесты)
